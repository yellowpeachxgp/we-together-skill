# We Together Identity 融合策略设计稿

## 1. 文档目标

本文档定义 `we together` 第一阶段的身份融合策略。

重点回答：

- 不同来源的账号、昵称、邮箱、手机号、别名如何自动对齐
- 激进自动融合如何保持可逆
- merge 与 split 的决策边界是什么
- 弱证据如何进入局部分支而不是直接污染主图谱

## 2. 设计原则

### 2.1 默认激进融合

第一阶段默认采用激进自动融合，因为碎裂图谱无法验证系统价值。

### 2.2 存储必须保守可逆

即使运行时采用激进融合，底层也必须保留：

- 原始身份
- 匹配依据
- 置信度
- 合并轨迹
- 拆分能力

### 2.3 弱证据不直接当最终真相

弱证据可以触发候选融合，但不应直接形成不可逆强结论。

### 2.4 多面人格不是身份分裂

同一个人在不同场景中的自我矛盾，应通过 `Person facet` 和 `Scene` 表达，不应误建成两个 `Person`。

## 3. 融合输入

身份融合模块的输入来自统一 importer 契约中的：

- `IdentityCandidate`
- `RelationClue`
- `EventCandidate`
- `FacetCandidate`
- `RawEvidence`

## 4. 匹配层级

### 4.1 强匹配

可作为直接高置信融合依据：

- 平台稳定账号 ID
- 邮箱
- 手机号
- 明确唯一 handle
- 同一系统中的不可重复用户标识

### 4.2 中匹配

需要结合上下文：

- 姓名
- 昵称
- 别名
- 职级与岗位
- 部门与组织位置
- 长期重复称呼

### 4.3 弱匹配

只可作为候选增强项：

- 语言风格相似
- 共现关系
- 同步时间线
- 相同群体高频出现
- 他人对其的称呼模式

## 5. 自动融合流程

### 5.1 标准流程

1. 导入器输出 `IdentityCandidate`
2. 归一化层做清洗与去重
3. 根据强、中、弱匹配规则计算候选分数
4. 形成若干 identity cluster
5. 对高置信 cluster 自动生成 `merge_entities` patch
6. 对中低置信 cluster：
   - 可以先链接到同一 `Person`
   - 或创建 `local_branch`
7. 更新 `IdentityLink`
8. 记录 merge ledger

### 5.2 Cluster 规则

第一阶段建议以“候选簇”为单位思考融合，而不是每次两两比较到底。

一个 cluster 可以包含：

- 微信昵称
- 飞书实名
- Slack handle
- 邮箱
- 某段口述里的代号

## 6. 评分模型

第一阶段推荐采用可解释打分，而不是黑盒模型直判。

建议分值来源：

- 强匹配分
- 中匹配分
- 关系上下文分
- 群体共现分
- 时间线连续性分
- 风格相似分
- 冲突惩罚分

### 6.1 冲突惩罚

若以下条件冲突，应明显降分：

- 同一时间明显同时出现在互斥场景
- 组织身份互相矛盾
- 平台稳定 ID 指向不同人
- 多个来源明确表明是不同个体

## 7. Merge 策略

### 7.1 自动 merge 条件

满足以下条件可直接自动 merge：

- 至少一个强匹配命中
- 或多个中匹配叠加到高置信阈值
- 且不存在强冲突证据

### 7.2 谨慎 merge 条件

以下情况允许暂时链接但不做最终强 merge：

- 无强匹配，仅有中匹配组合
- 关系上下文强，但身份字段弱
- 用户口述与外部数据之间存在别名重叠

### 7.3 不应直接 merge 的情况

- 仅有风格相似
- 仅有群体共现
- 同名但组织信息不稳定
- 同一别名指向多个人

## 8. Split 策略

### 8.1 触发条件

如果已合并对象后续出现以下问题，应允许 split：

- 新强证据显示不是同一个人
- 关系网络出现结构性矛盾
- 时间线明显互斥
- 多个平台 ID 彼此冲突

### 8.2 Split 处理

split 不是简单复制人物描述，而应：

- 重挂 `IdentityLink`
- 重挂相关 `Relation`
- 重挂相关 `Memory`
- 重建相关 `State`
- 记录 `split_entity` patch

## 9. Local Branch 策略

### 9.1 适用场景

局部分支优先用于：

- 身份是否应 merge
- 哪个别名属于谁
- 某条联系方式是否应挂到某个 `Person`

### 9.2 处理方式

当证据不足以形成高置信 merge 时：

- 创建 `local_branch`
- 为不同解释创建 `branch_candidate`
- 允许运行时暂时采用较高分解释
- 但存储层保留分歧

## 10. Merge Ledger

第一阶段建议显式保留 merge ledger。

用途：

- 记录每次 merge 的参与对象
- 记录 merge 原因
- 记录 merge 置信度
- 记录后续是否被 split

建议字段可落在：

- `patches`
- `local_branches`
- 或单独的 `merge_ledger` 表

第一阶段若控制复杂度，可先复用 `patches`。

## 11. 运行时与身份融合的关系

运行时可以先基于当前最佳 merge 结果工作，但必须保证：

- 运行时使用的 `Person` 仍可回溯到多个 `IdentityLink`
- 运行时不直接抹掉原始身份边界

## 12. 第一阶段最小实现要求

第一阶段至少要做到：

- 有强 / 中 / 弱匹配分层
- 有 cluster 级融合
- 有 merge patch
- 有 split patch
- 有 local branch 支持
- 有可回溯 identity -> person 映射

## 14. 结论

`we together` 的身份融合不能做成：

- 保守到把同一个人拆成四五个人
- 激进到把不同人永久合成一个人

它应被实现为：

> **默认激进融合、底层可逆留痕、弱证据局部分支、强证据驱动自动 merge 的身份对齐系统。**
